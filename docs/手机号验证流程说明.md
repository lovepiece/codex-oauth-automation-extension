# 手机号验证流程说明

本文档总结当前浏览器插件的手机号验证流程，包括触发入口、Hero-SMS 号码生命周期、页面交互、短信轮询、失败处理和号码释放策略。

## 1. 触发入口

手机号验证发生在自动流程的步骤 9。

步骤 9 会先检查当前认证页状态：

1. 如果检测到手机号页面，进入 Hero-SMS 自动接码流程。
2. 如果页面已经进入 OAuth 同意页，说明无需手机号验证，步骤 9 直接跳过。
3. 如果页面是认证重试页，步骤 9 记录后跳过。
4. 如果等待超时仍未检测到手机号页面，也按无需手机号验证处理并跳过。
5. 如果检测到 Cloudflare 安全防护阻断，会抛出安全阻断错误并停止流程。

检测手机号页时使用认证页内容脚本读取页面状态，主要关注：

| 状态 | 含义 |
| --- | --- |
| `addPhonePage` | 当前在手机号验证相关页面 |
| `phoneNumberEntryPage` | 当前在手机号填写页 |
| `phoneVerificationPage` | 当前在短信验证码填写页 |
| `hasPhoneInput` | 页面存在手机号输入框 |
| `hasCodeTarget` | 页面存在短信验证码输入框 |
| `hasActionButton` | 页面存在提交/继续按钮 |
| `hasResendButton` | 页面存在重发验证码按钮 |
| `hasRetryButton` | 页面存在重试按钮 |
| `errorText` | 页面展示的错误文案 |

## 2. 关键配置

当前手机号验证相关的核心参数：

| 配置 | 当前值 | 说明 |
| --- | --- | --- |
| 号码最大复用次数 | 5 次 | 同一个号码最多可成功验证 5 次 |
| 号码本地有效期 | 20 分钟 | 获取号码后本地认为 20 分钟内可复用 |
| 短信轮询间隔 | 5 秒 | 等待验证码时轮询 Hero-SMS 状态 |
| 短信等待超时 | 180 秒 | 单轮等待验证码最长 3 分钟 |
| Hero-SMS 重发间隔 | 60 秒 | 等待期间每 60 秒触发一次 `setStatus=3` |
| 手机号验证重试次数 | 3 次 | 遇到可换号错误时最多尝试 3 个号码 |
| 验证码被拒绝重试次数 | 3 次 | 验证码被页面拒绝时最多再请求 3 次新短信 |
| 失败号码清理延迟 | 2 分钟 | 失败列表里的号码延迟清理 |
| 备用号码重试延迟 | 5 分钟 | 释放失败但仍有剩余有效期的号码进入备用列表 |

## 3. 号码获取和复用策略

进入手机号页面后，系统会先确保有一个可提交的 Hero-SMS 激活号码。

号码选择顺序：

1. 优先复用当前本地保存的号码。
2. 当前号码不可复用时，先尝试释放旧号码。
3. 尝试从备用列表恢复可复用号码。
4. 尝试从 Hero-SMS 活动激活列表恢复远端仍活动的号码。
5. 如果都没有可用号码，则调用 Hero-SMS `getNumberV2` 获取新号码。

一个号码可复用必须同时满足：

| 条件 | 说明 |
| --- | --- |
| service 匹配 | 与当前配置的 Hero-SMS 服务编号一致 |
| country 匹配 | 与当前配置的国家 ID 一致 |
| useCount 小于 5 | 未达到最大成功使用次数 |
| 未过期 | 本地剩余有效期大于 0 |
| 未取消 | 最近状态不是 `STATUS_CANCEL` |
| 未释放 | 没有被标记为已释放 |

获取新号码时使用：

```text
action=getNumberV2
service=配置的服务编号
country=配置的国家 ID
maxPrice=配置的最高价格，可选
```

获取成功后保存：

| 字段 | 说明 |
| --- | --- |
| `activationId` | Hero-SMS 激活 ID |
| `phoneNumber` | 手机号 |
| `service` | 服务编号 |
| `country` | 国家 ID |
| `acquiredAt` | 获取时间 |
| `expiresAt` | 本地过期时间 |
| `useCount` | 已成功使用次数 |
| `resendCount` | 已请求重发次数 |
| `lastCode` | 最近收到的验证码 |
| `lastStatus` | 最近 Hero-SMS 状态 |

## 4. 提交手机号

拿到号码后，系统会再次同步 Hero-SMS 活动激活列表，确认当前号码仍存在于远端活动列表中。确认失败时会停止填写，避免提交已失效号码。

提交手机号流程：

1. 读取号码对应国家信息。
2. 等待手机号页面内容脚本就绪。
3. 等待手机号输入框出现。
4. 将完整国际格式手机号填入页面。
5. 如页面存在隐藏 `phoneNumber` 字段，也同步填入同一号码。
6. 等待提交/继续按钮可用。
7. 通过 React/ARIA 兼容点击方式提交手机号。
8. 等待页面进入短信验证码输入页。

如果提交后页面没有进入 `phone-verification` 验证码页，会读取当前页面状态做二次确认；仍没有验证码输入框时，视为提交失败。

## 5. 接收短信和回填验证码

手机号提交成功后，系统会先等待 5 秒，再调用 Hero-SMS `setStatus=3` 请求一条新的短信验证码。

随后进入等待验证码流程：

1. 默认先调用 `setStatus=1` 标记激活准备就绪。
2. 每 5 秒调用一次 `getStatusV2`。
3. 如果刚调用过 `setStatus=3` 请求新短信，先比较 `sms.dateTime` / 电话验证码时间是否相对上一条验证码更新。
4. 如果 `dateTime` 和上一条一致，说明 Hero-SMS 仍在返回旧短信，继续等待，不把当前 `code` 当作新验证码。
5. 如果 `dateTime` 已更新，再读取 `sms.code` 或电话验证码字段并返回给页面回填逻辑。
6. 如果验证码与已排除验证码重复，则继续等待。
7. 等待满 60 秒仍未收到新验证码时，触发一次重发。
8. 总等待时间超过 180 秒时，抛出 `hero_sms_wait_code_timeout`。

收到验证码后：

1. 保存最近验证码到当前激活记录。
2. 将验证码填入页面。
3. 如果页面自动提交，等待页面结果。
4. 如果页面没有自动提交，则点击提交按钮。
5. 等待页面离开手机号页，或确认进入下一阶段。

## 6. 重发策略

系统有两类重发动作。

Hero-SMS 侧重发：

```text
action=setStatus
status=3
```

页面侧重发：

```text
点击页面上的“重新发送验证码”按钮
```

具体策略：

1. 手机号提交成功后，先等待 5 秒，再通过 Hero-SMS 请求新短信。
2. 如果是复用号码，会额外等待约 3 秒后尝试点击页面的重发按钮。
3. 等待验证码期间，每 60 秒触发一次 Hero-SMS 重发。
4. 第一次等待满 60 秒时，会额外尝试点击页面重发按钮。
5. 如果页面重发按钮点击成功，会等待 5 秒后调用 Hero-SMS `setStatus=3`。
6. 如果页面重发按钮不可用，则继续等待 Hero-SMS 轮询。

## 7. 验证码被拒绝时的处理

如果验证码填入页面后，页面仍停留在手机号页并出现错误文案，视为验证码被拒绝。

处理流程：

1. 将被拒绝的验证码加入排除集合。
2. 不重新提交手机号。
3. 调用 Hero-SMS `setStatus=3` 请求不同的新短信。
4. 再次轮询 `getStatusV2`。
5. 轮询时先要求 `dateTime` 相对上一次短信更新；如果 `dateTime` 没变，即使 `code` 字段存在也仍按旧短信处理。
6. 如果 `dateTime` 已更新，但新验证码和上一次被拒绝的验证码相同，不填入页面，继续调用 `setStatus=3` 等待新验证码。
7. 如果 `dateTime` 已更新且验证码和上一次不同，则填入页面。
8. 连续 3 次仍拿到相同或无效验证码后，才更换号码。

超过重试次数后，会把错误抛给外层失败处理，并按失败号码策略处理当前号码。

## 8. 成功后的处理

手机号验证码提交成功后，会记录一次手机号验证成功。

处理规则：

1. 当前号码 `useCount + 1`。
2. 清空 `heroSmsPendingSuccessActivationId`。
3. 如果号码未达到最大复用次数且未过期，继续保留当前号码供后续复用。
4. 如果号码达到最大复用次数，调用 Hero-SMS `setStatus=6` 完成激活并释放。
5. 如果号码已过期，调用 Hero-SMS `setStatus=8` 取消并释放。

完整账号流程最终成功后，还会再次执行成功收尾：

1. 不再增加使用计数，因为手机号验证码通过时已经计数。
2. 如果当前号码仍是待成功确认的号码，只根据现有 `useCount` 和有效期判断是否释放。
3. 达到最大使用次数时完成释放。
4. 未达到最大使用次数时继续保留复用。

## 9. 失败分类

手机号验证失败后，会先把错误归类。

| 失败原因 | 触发条件 |
| --- | --- |
| `hero_sms_wait_code_timeout` | 等待 Hero-SMS 验证码超过 180 秒 |
| `phone_max_usage_exceeded` | 页面提示 `phone_max_usage_exceeded` |
| `phone_resend_rate_limited` | 页面提示重发次数过多 |
| `phone_sms_unavailable` | 页面提示当前号码无法接收短信 |
| `phone_verification_invalid_code` | 页面拒绝验证码 |
| `phone_verification_failed` | 其他手机号验证错误 |

其中以下错误会触发换新号码：

| 错误 | 是否换号 |
| --- | --- |
| `hero_sms_wait_code_timeout` | 是 |
| `phone_max_usage_exceeded` | 是 |
| `phone_resend_rate_limited` | 是 |
| `phone_sms_unavailable` | 是 |

验证码被拒绝不会立即换号，也不会重新提交手机号；它会优先对同一号码请求新短信，并要求新验证码必须和上一次不同。连续 3 次仍无法拿到不同的新验证码后，才向外抛错并进入换号流程。

## 10. 换号和页面恢复策略

遇到需要换号的错误后，系统会处理当前号码，并尝试让页面回到手机号填写状态。

号码处理：

| 错误 | 号码处理 |
| --- | --- |
| 提交手机号后触发 `phone_max_usage_exceeded` | 优先 `setStatus=6` 完成激活 |
| 非提交阶段触发 `phone_max_usage_exceeded` | 等待 2 分钟后 `setStatus=8` 取消 |
| 重发次数过多 | 等待 2 分钟后 `setStatus=8` 取消 |
| 无法接收短信 | 立即加入失败列表，后台 2 分钟后 `setStatus=8` 取消 |
| 验证码错误超过重试次数 | 等待 2 分钟后 `setStatus=8` 取消 |
| 其他手机号验证失败 | 等待 2 分钟后 `setStatus=8` 取消 |
| 等待短信超时 | 等待 2 分钟后 `setStatus=8` 取消 |

`setStatus=8` 成功时应返回：

```text
ACCESS_CANCEL
```

如果取消失败并返回 HTTP 409，例如：

```json
{
  "title": "EARLY_CANCEL_DENIED",
  "details": "Activation cannot be cancelled at this time. Minimum activation period must pass.",
  "info": {
    "minActivationTime": 120
  }
}
```

或：

```json
{
  "title": "OTP_RECEIVED",
  "details": "Cannot terminate activation - OTP has been received on this number"
}
```

则不继续强行取消，当前号码会加入备选池，后续再按备选池规则重试复用或释放。

页面恢复：

| 错误 | 页面恢复动作 |
| --- | --- |
| `phone_max_usage_exceeded` | 点击页面“重试”按钮 |
| `phone_resend_rate_limited` | 浏览器历史后退回手机号填写页 |
| `hero_sms_wait_code_timeout` | 浏览器历史后退回手机号填写页，然后抛出重启步骤 7 错误 |
| `phone_sms_unavailable` | 浏览器历史后退回手机号填写页 |

换号最多尝试 3 轮。超过 3 轮后，流程抛出失败：

```text
连续 3 次触发手机号不可用错误，未能申请到可用手机号。
```

## 11. 等待验证码超时的特殊处理

等待 Hero-SMS 短信超过 180 秒时，会抛出 `hero_sms_wait_code_timeout`。

该错误会触发：

1. 当前号码按失败原因处理，等待 2 分钟后尝试 `setStatus=8` 取消。
2. 页面尝试后退回手机号填写页。
3. 抛出 `hero_sms_timeout_restart_step7`。
4. 外层自动流程会回到步骤 7，重新走后续 OAuth/手机号流程并申请新号码。

这样做是为了避免认证页状态卡死在旧的手机号验证码页。

## 12. 号码释放策略

释放号码时优先使用 Hero-SMS `setStatus`：

| 释放模式 | status | 说明 |
| --- | --- | --- |
| cancel | 8 | 取消激活 |
| complete | 6 | 完成激活 |

如果 `setStatus` 失败：

| 模式 | 备用接口 |
| --- | --- |
| complete | `finishActivation` |
| cancel | `cancelActivation` |

释放成功后：

1. 如果释放的是当前号码，清空当前号码。
2. 清空最近验证码。
3. 同步活动激活列表。

释放失败时：

1. 如果号码剩余有效期大于 5 分钟，移入备用列表，5 分钟后可再次尝试复用或释放。
2. 如果剩余有效期不足 5 分钟，保留失败记录等待后续清理。
3. 失败列表里的号码会在 2 分钟后由 alarm 清理。

## 13. 备用列表和失败列表

备用列表用于暂时不能释放但仍可能复用的号码。

备用号码可恢复条件：

| 条件 | 说明 |
| --- | --- |
| service/country 匹配 | 与当前配置一致 |
| 未达到最大复用次数 | `useCount < 5` |
| 未过期 | 剩余有效期大于 0 |
| 已到重试时间 | `retryAt <= now` |
| 未取消 | 最近状态不是 `STATUS_CANCEL` |

失败列表用于等待清理的号码。

失败列表清理规则：

1. 到达 `cleanupAt` 后触发清理。
2. 如果号码因达到最大使用次数或 `phone_max_usage_exceeded` 失败，优先完成激活。
3. 其他失败优先取消激活。
4. 清理成功后记录 `completed` 或 `cancelled`。
5. 清理失败后记录 `cleanup_failed` 和错误信息。

## 14. 当前流程的主要 Hero-SMS API

| 场景 | API |
| --- | --- |
| 获取新号码 | `getNumberV2` |
| 查询活动激活列表 | `getActiveActivations` |
| 查询短信状态 | `getStatusV2` |
| 标记准备接收短信 | `setStatus&id=...&status=1` |
| 请求新短信/重发 | `setStatus&id=...&status=3` |
| 完成激活 | `setStatus&id=...&status=6` |
| 取消激活 | `setStatus&id=...&status=8` |
| 完成激活备用接口 | `finishActivation` |
| 取消激活备用接口 | `cancelActivation` |

## 15. 高层时序

完整成功路径：

```text
步骤 9 检测手机号页
  -> 确保 Hero-SMS 配置有效
  -> 复用当前号码 / 备用号码 / 远端活动号码 / 获取新号码
  -> getStatusV2 确认号码未取消
  -> getActiveActivations 确认号码仍活动
  -> 填写并提交手机号
  -> 等待 5 秒
  -> setStatus=3 请求新短信
  -> waitForCode 轮询 getStatusV2
  -> 收到验证码
  -> 填写验证码
  -> 页面离开手机号页
  -> 记录成功次数
  -> 根据次数和有效期决定保留或释放号码
  -> 步骤 9 完成
```

典型失败换号路径：

```text
步骤 9 检测手机号页
  -> 获取/复用号码
  -> 提交手机号或等待短信
  -> 页面提示号码不可用 / 重发过多 / 等待短信超时
  -> 释放或记录当前号码
  -> 页面恢复到手机号填写页
  -> 尝试下一个号码
  -> 最多 3 轮
```

验证码被拒绝路径：

```text
收到验证码
  -> 填入页面
  -> 页面提示验证码错误
  -> 记录该验证码为已拒绝
  -> 不重新提交手机号
  -> setStatus=3 请求新短信
  -> 排除旧验证码继续轮询
  -> 最多重试 3 次
```
